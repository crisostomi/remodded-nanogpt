remodded-nanogpt · Status Report

What this project is, why "render all 171 historical genes" is not the target, what's blocking it, and exactly where we stand today. Companion to the visual Gene Map.

In one paragraph. This repo is a search engine for the modded-nanogpt speedrun. A record is a set of small composable changes called genes; we pick any valid combination, auto-generate a runnable training script, and search for the next record (#84). We have 24 genes and an engine that renders the current record byte-for-byte. "Coverage" (today 58 / 171) measures how much of the historical gene-set our single-substrate engine can render — it is a progress gauge, not a goal to hit 100%. Most of the uncovered 113 are not missing toggles: they are whole different model scaffolds (substrates) our generator can't emit, or tuning knobs we deliberately model as config instead of genes.

1 The goal what success looks like

✅ What we ARE building

A gene-combinatorial search engine. Pick any valid subset of genes → generate a static, fast, torch.compile-friendly train_gpt.py → measure → search for a new record (#84). Genes are resolved at generation time, so there's no runtime branching in the hot path.

🚫 What we are NOT building

A reproduction of the 83 past records. Those are known-good reference points and a quarry of genes to recombine — not the deliverable. Reproducing all of them would need a fundamentally bigger generator (see Problem 1) and is explicitly out of scope.

The engine sits on one substrate — era E15, the current record's scaffold. Every gene is an E15-compatible change. "Render" means: emit a training script for a chosen gene combination, on that one scaffold.

3 The problems why coverage is 58, not 171

58 covered (34%)
113 uncovered

Covered = 50 via registered genes + 8 via the searchable TRAINING_STAGES curriculum dimension.

The 113 uncovered gene-rows are not "toggles we forgot to write." They fall into three buckets, plus one cross-cutting constraint (the no-GPU gap). The bottleneck is the single substrate, not effort.

Bucket of the 113 uncovered~sizeWhy it is not "just add a toggle"
Substrate rewrites
eras E0–E14		 ~26 A substrate is a whole different model / optimizer / forward scaffold, not a feature. Our generator emits exactly one scaffold (E15). Covering these means the generator must be able to emit an older scaffold — a multi-substrate / era-aware generator, which is a much larger architecture change. This is the hard ceiling.
Tuning genes
LR, betas, iters, version bumps		 ~11+ Deliberately not modelled as genes. They are searchable config dimensions (optim.* overrides + the TRAINING_STAGES curriculum). They're already covered by search; turning them into genes would be redundant — exactly why we just retired the 3 schedule genes.
Early-era alleles & additives
the rest		 ~rest They assume an older scaffold (pre-value-embeds, pre-MUDD, pre-bigram, FlexAttention, merged-QKV…). Porting each forward to E15 is per-gene work; some are superseded (FlexAttention was replaced by FlashAttention-3, which the code already uses); the optimizer/kernel ones need GPU validation. newton_schulz was one that ported cleanly (+1).

⚠️ Cross-cutting: the no-GPU validation gap

This dev box has no GPU and no torch. Architecture toggles are fully verified by compile + byte-equivalence + greps. But for optimizer- / loss-internal numerics (cautious-WD "off", newton_schulz), a numeric bug wouldn't surface until a real 8×H100 run — those are implemented faithfully but flagged GPU-pending.

✅ Already unblocked: faithful sourcing

The historical record source is reachable now (verified byte-identical to our baseline), so we can harvest real implementations instead of guessing. That removed the fidelity problem — but not the substrate problem above. Harvesting helps only for genes that compose with E15.
